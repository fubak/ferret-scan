Technical Field

15 The present invention relates generally to computer security and artificial intelligence infrastructure governance, and more particularly to automated risk assessment systems and methods that apply multi-dimensional scoring to validate Model Context Protocol (MCP) server configuration entries declared within AI agent deployment environments.

20 More specifically, the invention concerns a composite risk scoring framework that evaluates MCP server configurations across six independent analytical dimensions — command safety, environment variable exposure, source trustworthiness, naming reputation, permission scope breadth, and metadata integrity — and aggregates the resulting dimension scores into a single weighted risk value that drives automated BLOCK, WARN, or ALLOW recommendations.

Background of the Invention

25 The Model Context Protocol (MCP) is an open standard that enables AI assistants and agent frameworks to communicate with external servers that expose tools, resources, and prompts through a structured interface. AI coding assistant platforms such as Claude Code (Anthropic), Cursor, Windsurf, and Continue allow developers and enterprises to declare one or more MCP servers in project-level or user-level configuration files, typically formatted as JSON objects containing a server name,

30 an executable command, a set of command-line arguments, and a collection of environment variable assignments. Once declared, an MCP server is launched as a subprocess by the AI agent whenever the agent requires access to the capabilities that server exposes, which may include file system manipulation, network requests, code execution, database queries, external API invocation, and other high-privilege operations on behalf of the AI agent and its user.

35 The proliferation of third-party MCP servers presents significant supply-chain and runtime security risks. A malicious or misconfigured MCP server entry can cause an AI agent to execute arbitrary binaries, transmit sensitive environment variables to remote endpoints, expand its effective permission surface beyond what the user intended, or invoke tools whose names appear benign but whose implementations perform harmful operations. Unlike traditional software dependencies, MCP server

40 declarations do not benefit from established software composition analysis (SCA) pipelines, package signing infrastructure, or vulnerability database coverage. The identity and provenance of an MCP server are encoded in free-form string fields — command paths, argument arrays, and environment variable maps — that resist the structured parsing assumptions underlying conventional dependency auditing tools. Organizations deploying AI agents in enterprise settings therefore

45 lack systematic, automated mechanisms to assess the trustworthiness of MCP server configurations prior to permitting their execution. Existing static analysis tools for AI configuration files either detect only a narrow class of obvious vulnerabilities — such as hardcoded credential strings — or treat MCP server entries as opaque blobs that require manual human review. No prior art system applies a comprehensive multi-dimensional scoring model to MCP server validation that simultaneously

50 accounts for the command executable identity, argument structure, environment variable sensitivity, publisher reputation, naming similarity to known-good packages, requested permission breadth, and configuration completeness. There therefore exists an unmet need for an automated system and method that applies multi-dimensional risk scoring to MCP server configuration entries and produces actionable, human-readable recommendations that security teams and AI agent runtimes can consume without requiring manual inspection of individual configuration fields.

Summary of the Invention

55 The present invention provides a system and method for multi-dimensional risk scoring and validation of Model Context Protocol server configuration entries. In a first aspect, the invention provides a computer-implemented method comprising: receiving a structured MCP server configuration entry (400) that includes at minimum a command field (400a), an arguments array (400b), and an environment variable map (400c); evaluating the configuration entry across six independent risk dimensions by

60 applying, respectively, a command safety analyzer (402), an environment variable analyzer (404), a trusted source checker (406), a naming reputation scorer (408), a permission scope analyzer (410), and a metadata validator (412); computing a weighted composite risk score R through a composite risk calculator (414) according to the formula R = Σ(wi × Di) for i = 1 to 6, where each Di is the normalized score produced by dimension analyzer i

65 and each wi is a configurable weight coefficient assigned to that dimension; and generating, through a recommendation engine (416), an output disposition (420) of BLOCK, WARN, or ALLOW based on the composite risk score R relative to configurable low and high threshold values, together with a human-readable explanation identifying which specific configuration fields contributed most significantly to the score.

70 In a second aspect, the invention provides a system comprising a plurality of dimension-specific analyzer modules, a trusted source registry (418) organized into a hierarchical tier structure, a composite risk calculator (414), and a recommendation engine (416) that produces structured output dispositions. In a third aspect, the invention provides a non-transitory computer-readable medium storing instructions that, when executed by a processor, implement the multi-dimensional risk scoring method.

75 A key advantage of the present invention is that it operates entirely through static analysis of the configuration entry's string fields, without launching or communicating with the MCP server process, thereby avoiding the execution of potentially malicious code during the validation process itself. Another advantage is that the weighted scoring architecture permits security administrators to tune the relative importance of each dimension to match organizational risk tolerance, compliance requirements, or deployment context.

Detailed Description of the Preferred Embodiments

110 Reference will now be made in detail to the presently preferred embodiments of the invention, examples of which are illustrated in the accompanying drawings. Wherever possible, the same reference numbers will be used throughout the drawings to refer to the same or like parts. The term "MCP server configuration entry" as used herein refers to a structured data object, typically expressed as a JSON object, that declares all information necessary for an AI agent runtime to launch and communicate

115 with a Model Context Protocol server subprocess. The term "risk score" as used herein refers to a normalized numeric value in the range [0.0, 1.0] where 0.0 represents minimal risk and 1.0 represents maximum assessed risk. The term "disposition" as used herein refers to a categorical recommendation — BLOCK, WARN, or ALLOW — that an AI agent runtime or security enforcement layer may act upon.

I. System Architecture Overview (FIG. 1)

120 Referring now to FIG. 1, the multi-dimensional MCP server risk scoring system receives as its primary input an MCP server configuration entry (400). The configuration entry (400) is a structured record comprising at least three sub-fields: a command field (400a) specifying the executable path or command name used to launch the MCP server process; an arguments array (400b) containing zero or more string arguments passed to the command at launch time; and an

125 environment variable map (400c) comprising zero or more key-value pairs that are injected into the MCP server process environment at launch time. In typical deployments, additional optional fields including a display name, a version string, a description, a publisher identifier, and a list of declared tool permissions may also be present in the configuration entry (400) and are consumed by the relevant dimension analyzers as described below.

130 The configuration entry (400) is routed in parallel to six dimension-specific analyzer modules. Each analyzer module independently inspects one or more fields of the configuration entry (400), applies domain-specific heuristics, pattern matching, registry lookups, or structural analysis, and produces a normalized dimension score Di in the range [0.0, 1.0] representing the risk contribution of that dimension. The six dimension analyzers are:

135 the command safety analyzer (402), the environment variable analyzer (404), the trusted source checker (406), the naming reputation scorer (408), the permission scope analyzer (410), and the metadata validator (412). Each of these modules is described in detail in Section II below. The dimension scores D1 through D6 produced by the analyzers are transmitted to the composite risk calculator (414).

140 The composite risk calculator (414) applies a configurable weighted summation formula to produce a single composite risk score R. The formula is:

R = Σ(wi × Di) for i = 1..6

145 where wi are non-negative weight coefficients satisfying Σwi = 1.0, and Di are the normalized dimension scores. The weight coefficients are stored in a configurable policy object that security administrators may adjust to reflect organizational priorities. Default weight assignments in a preferred embodiment allocate the largest weights to the command safety dimension (D1) and the trusted source

150 dimension (D3), reflecting the empirical finding that unknown or suspicious executables and unregistered sources represent the highest-probability vectors for supply-chain compromise of AI agent environments. The composite risk calculator (414) also consults the trusted source registry (418) during score computation when evaluating the source trust component, as further described in Section III below.

155 The composite risk score R is passed to the recommendation engine (416), which compares R to a configurable low threshold Tlow and a configurable high threshold Thigh, where 0.0 < Tlow < Thigh < 1.0. If R < Tlow, the recommendation engine (416) produces an ALLOW disposition. If Tlow ≤ R < Thigh, the recommendation engine (416) produces a WARN disposition. If R ≥ Thigh,

160 the recommendation engine (416) produces a BLOCK disposition. In all cases, the recommendation engine (416) also generates a human-readable explanation that identifies the specific configuration fields that contributed most significantly to the composite score, the dimension scores that exceeded configurable per-dimension alert thresholds, and any specific heuristic rules or registry lookups that were triggered during analysis. The resulting output (420) comprises the disposition value and the explanation text.

II. Dimension Analyzer Modules

165 The command safety analyzer (402) operates on the command field (400a) of the configuration entry (400). In a preferred embodiment, the command safety analyzer (402) maintains an internal blocklist of known high-risk executable names and path patterns, including but not limited to shell interpreter invocations (e.g., bash, sh, zsh, cmd.exe, powershell), scripting language runtimes invoked with inline execution flags (e.g., python -c, node -e, perl -e), and path

170 patterns indicative of temporary or user-writable directories (e.g., /tmp/, %TEMP%, ~/.cache/). The command safety analyzer (402) also evaluates whether the command field (400a) contains an absolute path versus a bare executable name, assigning elevated risk to bare names that rely on PATH resolution, which is susceptible to PATH hijacking attacks. Commands that reference package-manager-scoped executables — for example, commands of the form

175 "npx <package-name>" or "uvx <package-name>" — are recognized by the command safety analyzer (402) as delegating execution to a package registry resolver, causing the extracted package name to be forwarded to the trusted source checker (406) and naming reputation scorer (408) as an additional evaluation target supplementing the command field (400a) itself. The arguments array (400b) is also scanned for shell metacharacters, command

180 substitution patterns, and suspicious flag combinations. The dimension score D1 produced by the command safety analyzer (402) is a function of the number and severity of heuristic rule matches across both the command field (400a) and the arguments array (400b), normalized to [0.0, 1.0].

185 The environment variable analyzer (404) operates on the environment variable map (400c). It applies a multi-stage inspection procedure. First, each key in the map is evaluated against a pattern library of sensitive variable name patterns, including patterns associated with cloud provider credentials (e.g., AWS_ACCESS_KEY_ID, AZURE_CLIENT_SECRET, GOOGLE_APPLICATION_CREDENTIALS), version control platform tokens (e.g., GITHUB_TOKEN, GITLAB_TOKEN),

190 container registry credentials, database connection strings, and generic secret indicators (e.g., variables whose names contain the substrings SECRET, PASSWORD, TOKEN, KEY, or CREDENTIAL). Second, each value in the map is evaluated to determine whether it contains a literal credential string — identified by entropy analysis, known credential format patterns (e.g., Bearer tokens, API key formats), or the presence of known-secret prefixes —

195 as opposed to a shell variable reference (e.g., ${MY_SECRET}) that defers value resolution to the runtime environment. Configurations that pass sensitive variable names with literal values are assigned higher risk than those that use shell variable references, since the former embed secrets directly in the configuration file. The dimension score D2 is a weighted function of the count of sensitive variable name matches and the count of literal credential value detections.

200 The trusted source checker (406) evaluates the provenance of the MCP server by consulting the trusted source registry (418). In a preferred embodiment, the trusted source checker (406) extracts a source identifier from the configuration entry (400), which may be a package name derived from the command field (400a) (e.g., the package name argument to npx or uvx), a publisher name or organization identifier present in an optional metadata

205 field of the configuration entry (400), or a repository URL present in the arguments array (400b). The extracted source identifier is looked up in the trusted source registry (418). If the source identifier matches a Tier 1 entry (454) in the registry, indicating an official platform publisher, the dimension score D3 is assigned a near-zero value reflecting minimal source-provenance risk. If the source identifier matches a Tier 2

210 entry (456), indicating a verified community publisher, D3 is assigned a low but non-zero value. If the source identifier matches a Tier 3 entry (458), indicating an individually reviewed package that has not achieved full community verification, D3 is assigned a moderate value. If no match is found in any tier, the source is classified as unregistered (460) and D3 is assigned a high value. Additionally, the trusted

215 source checker (406) invokes the typosquatting detection module (462) when the source identifier does not match any registry entry, as further described in Section III below.

220 The naming reputation scorer (408) evaluates the textual characteristics of the MCP server name and any package identifiers present in the configuration entry (400) using pattern recognition heuristics informed by known naming patterns of malicious packages documented in public supply-chain security incident databases. The naming reputation scorer (408) applies the following sub-analyses: (a) common-word impersonation detection, which identifies

225 names that substitute common words for security-sensitive brand names (e.g., "mcp-server-cIaude" using a capital I in place of lowercase L); (b) version manipulation detection, which identifies names appended with version-like suffixes that differ from the canonical package's versioning scheme; (c) category mismatch detection, which flags packages whose declared tool capabilities do not match the semantic domain implied by their name; and

230 (d) excessive scope signaling, which identifies names containing high-privilege capability keywords (e.g., "admin", "root", "system", "full-access") that are inconsistent with the declared functional purpose of the server. The dimension score D4 produced by the naming reputation scorer (408) is a function of the number and severity of naming anomaly detections, normalized to [0.0, 1.0].

II. Dimension Analyzer Modules (continued)

235 The permission scope analyzer (410) evaluates the breadth of capabilities that the MCP server declares itself to require, or that can be inferred from the tools and resources it exposes. In embodiments where the configuration entry (400) includes an explicit permission manifest listing tool categories or resource access types, the permission scope analyzer (410) maps the declared permissions to a capability taxonomy and assigns

240 risk weights to each category, with higher weights assigned to capabilities that enable file system modification, code execution, network egress, and credential access. In embodiments where no explicit permission manifest is present, the permission scope analyzer (410) infers likely capabilities from the executable name, known tool declarations for recognized packages in the trusted source registry (418), and any descriptive metadata fields

245 present in the configuration entry (400). The dimension score D5 is computed as a weighted sum of the risk weights of all detected or inferred capability categories, normalized to [0.0, 1.0] using a saturation function that prevents any single high-risk capability from driving the dimension score to its maximum while suppressing all other signals.

250 The metadata validator (412) evaluates the structural completeness and internal consistency of the configuration entry (400). Indicators evaluated by the metadata validator (412) include: (a) absence of a human-readable description field, which may indicate a programmatically generated or obfuscated entry; (b) absence of a version field or use of a wildcard version specifier, which prevents reproducible builds and complicates

255 security auditing; (c) mismatch between a declared display name and the command or package name, which may indicate a deceptive configuration; (d) presence of arguments or environment variables whose keys or values contain non-printable characters or unusual Unicode codepoints, which may indicate obfuscation of malicious payloads; and (e) structural anomalies such as duplicate keys, excessively long string values, or deeply nested

260 JSON structures that deviate from the expected flat schema of a valid MCP server configuration entry. The dimension score D6 is a function of the count and severity of metadata anomalies detected, normalized to [0.0, 1.0]. A configuration entry with no metadata anomalies receives D6 = 0.0. Each detected anomaly increases D6 by a proportional amount, with the most severe anomalies capable of driving D6 to its maximum value independently.

III. Trusted Source Registry and Typosquatting Detection (FIG. 3)

265 Referring now to FIG. 3, the trusted source registry (418) is a structured data store that organizes known MCP server publishers and packages into a four-level hierarchy. Tier 1 (454) — Official Platform Publishers — comprises packages and publishers that are directly maintained or formally endorsed by the organizations that develop the primary AI agent platforms, such as official MCP server packages published under the verified namespaces

270 of AI platform vendors. Tier 1 (454) entries receive the highest trust score and the lowest source-dimension risk contribution D3. Tier 2 (456) — Verified Community Publishers — comprises packages and publishers that have undergone a formal verification process by the operator of the trusted source registry (418), including review of source code, security disclosure history, and organizational identity confirmation. Tier 2 (456) entries

275 receive moderate trust and a low-to-moderate D3 contribution. Tier 3 (458) — Individually Reviewed Packages — comprises packages that have been reviewed on a per-package basis without full publisher-level verification, typically through community-submitted review processes or automated static analysis with human confirmation. Tier 3 (458) entries receive limited trust and a moderate D3 contribution. Packages

280 and publishers that do not appear in any tier of the trusted source registry (418) are classified as unregistered sources (460) and receive the highest D3 contribution, reflecting the absence of any independent verification of their provenance or safety.

285 The typosquatting detection module (462) is invoked by the trusted source checker (406) whenever a source identifier is classified as an unregistered source (460). The typosquatting detection module (462) computes the edit distance — using a combination of Levenshtein distance, Jaro-Winkler similarity, and keyboard-adjacency-weighted distance metrics — between the unregistered source identifier and each entry in the Tier 1 (454) and

290 Tier 2 (456) portions of the trusted source registry (418). If any comparison yields a similarity score above a configurable typosquatting detection threshold, the typosquatting detection module (462) raises a suspicious flag (464) and associates the unregistered source identifier with the most similar registered entry as a potential impersonation target. The suspicious flag (464) has two effects: it causes the source-dimension

295 score D3 to be assigned its maximum value regardless of other registry lookup results, and it causes the recommendation engine (416) to include a specific typosquatting warning in the human-readable explanation generated in the output (420), naming both the suspicious unregistered source identifier and the registered entry it most closely resembles.

IV. Method of Operation (FIG. 2)

300 Referring now to FIG. 2, the method of multi-dimensional MCP server risk scoring begins at START (422). In step 424, the system receives an MCP server configuration entry (400) from one of several possible sources: a configuration file parser that has read and deserialized an MCP configuration file from a project directory or user home directory; an API endpoint through which a CI/CD pipeline or deployment management system submits a configuration

305 entry for pre-deployment validation; or an interactive configuration editor that submits entries for real-time risk feedback. The received configuration entry (400) is deserialized into an internal representation that exposes the command field (400a), arguments array (400b), and environment variable map (400c) as typed fields accessible to all downstream analyzer modules.

310 In step 426, the command safety analyzer (402) evaluates the configuration entry (400) to produce dimension score D1, as described in Section II above. In step 428, the environment variable analyzer (404) evaluates the environment variable map (400c) to produce dimension score D2. In step 430, the trusted source checker (406) consults the trusted source registry (418) to produce dimension score D3,

315 invoking the typosquatting detection module (462) as necessary. In step 432, the naming reputation scorer (408) evaluates the package and server name fields to produce dimension score D4. In step 434, the permission scope analyzer (410) evaluates declared or inferred capabilities to produce dimension score D5. In step 436, the metadata validator (412) evaluates structural completeness and consistency to produce dimension score D6.

320 In step 438, the composite risk calculator (414) applies the weighted summation formula R = Σ(wi × Di) for i = 1..6 to produce the composite risk score R. In step 440, the recommendation engine (416) tests whether R is less than the low threshold Tlow. If yes, the method proceeds to step 442 where an ALLOW disposition is generated, then to step 450. If no, the method proceeds to

325 step 444, where the recommendation engine (416) tests whether R is less than the high threshold Thigh. If yes, the method proceeds to step 446 where a WARN disposition is generated, then to step 450. If no, the method proceeds to step 448 where a BLOCK disposition is generated, then to step 450. In step 450, the recommendation engine (416) generates a human-readable explanation and the output (420) comprising the disposition and explanation is emitted. The method then terminates.

Abstract

540 A system and method for multi-dimensional risk scoring of Model Context Protocol (MCP) server configuration entries in AI agent environments. A configuration entry comprising a command field, arguments array, and environment variable map is evaluated across six independent dimensions by parallel analyzer modules: a command safety analyzer, an environment variable analyzer, a trusted source checker, a naming reputation scorer, a permission scope analyzer, and a metadata validator. Each analyzer produces a normalized dimension score, and a composite risk calculator aggregates the scores using a weighted summation formula R = Σ(wi × Di). A recommendation engine compares the composite score to configurable thresholds and emits a BLOCK, WARN, or ALLOW disposition together with a human-readable explanation. A hierarchical trusted source registry and a typosquatting detection module provide provenance verification. The system operates entirely through static analysis, without executing the MCP server process.
