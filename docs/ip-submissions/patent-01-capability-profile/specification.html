Technical Field

15 The present invention relates generally to artificial intelligence security and software composition analysis, and more particularly to automated static analysis systems and methods that generate structured capability profiles from configuration files associated with AI coding assistant command-line interface (CLI) tools.

20 More specifically, the invention concerns the detection, classification, and risk assessment of AI agent capabilities declared in configuration files such as CLAUDE.md, .mcp.json, settings.json, and equivalent platform-specific formats, enabling enterprise security teams to systematically audit the capability footprint of AI CLI deployments within software development environments.

Background of the Invention

25 AI coding assistants have become pervasive in modern software development workflows. Tools such as Claude Code (Anthropic), Cursor, Windsurf, Continue, Aider, Cline, and related platforms allow individual developers to delegate complex coding, file manipulation, shell execution, network access, and third-party tool invocation to AI agents operating within the developer's local environment or cloud-connected infrastructure.

30 Each of these platforms exposes its permitted capabilities through one or more configuration files stored in the project directory or user home directory. For example, Claude Code reads CLAUDE.md files to determine behavioral rules and tool allowances; Model Context Protocol (MCP) servers are declared in .mcp.json files that specify the external processes an agent may invoke; agent behavioral rules, allowed tools, and permission overrides are stored in

35 settings.json files. Despite the proliferation of these capability-defining configuration files, no prior art system has systematically and automatically classified the capabilities declared across multiple AI CLI platforms using a unified canonical taxonomy.

40 Traditional software composition analysis (SCA) tools operate on dependency manifests such as package.json, requirements.txt, or go.mod, identifying known vulnerable library versions through comparison against vulnerability databases. These tools are not designed to parse free-form configuration files, interpret semantic content within Markdown-formatted behavioral instructions, or reason about capability declarations that are expressed through

45 tool names and natural-language permission grants rather than formal import statements or API call graphs. Consequently, organizations deploying AI CLI tools in enterprise environments currently lack automated mechanisms to understand, audit, or enforce capability boundaries for those tools.

50 There is therefore a need for a system and method capable of detecting which AI CLI platform is in use from the presence and structure of configuration files, parsing those files according to their platform-specific schemas, extracting and normalizing capability declarations into a platform-independent canonical representation, assigning risk levels to individual capabilities and aggregating those levels into an overall risk score, and producing structured reports in formats compatible with existing security toolchains.

Summary of the Invention

55 The present invention provides a system and method for automatically generating a structured AI agent capability profile from one or more AI CLI configuration files. The system comprises an agent detection module that identifies the AI CLI platform from the presence, naming, and structural characteristics of configuration files found in a target directory; a configuration parser that reads and interprets the contents of those files according to

60 platform-specific schemas; a capability extractor that maps parsed declarations to a canonical set of eighteen capability types organized in a four-tier risk taxonomy; a risk assessment engine that assigns base risk levels to individual capabilities and computes an aggregate risk score through a monotonically escalating combination function; and a report generator that produces output in SARIF, HTML, and JSON formats.

65 In a preferred embodiment, the invention operates without executing any of the AI CLI tools being analyzed, relying entirely on static inspection of configuration file content. The method is platform-agnostic at its core, supporting at least seven distinct AI CLI platforms through a registry mechanism that may be extended to accommodate future platforms without modification of the core capability extraction or risk assessment logic.

Detailed Description of the Preferred Embodiments

90 Reference will now be made in detail to the presently preferred embodiments of the invention, examples of which are illustrated in the accompanying drawings. Wherever possible, the same reference numbers will be used throughout the drawings to refer to the same or like parts.

I. System Architecture Overview (FIG. 1)

95 Referring now to FIG. 1, the AI agent capability profile generation system comprises a horizontal processing pipeline that accepts a configuration file input (100) at its left terminus and produces a capability profile (114) at its right terminus, with auxiliary inputs feeding into the pipeline from a supporting subsystem layer positioned below the main pipeline. The pipeline components are, from left to right: agent detection module

100 (102), configuration parser (104), capability extractor (106), and risk assessment engine (108). The capability profile (114) produced by the risk assessment engine (108) is subsequently consumed by a report generator (116) that produces human-readable and machine-readable output artifacts.

105 Two auxiliary inputs feed into the pipeline from the supporting subsystem layer. The AI CLI registry (110) provides platform identification data and tool-name mappings to both the agent detection module (102) and the capability extractor (106). The capability taxonomy (112) provides the authoritative mapping from capability types to risk tiers and is consumed exclusively by the risk assessment engine (108).

II. Agent Detection Module (102)

110 The agent detection module (102) receives the configuration file input (100), which may comprise a file path, a directory path, or a set of file paths provided by a caller. The module queries the AI CLI registry (110) to obtain a list of known platform entries, each of which specifies one or more file-name glob patterns characteristic of that platform's configuration files. For example, the Claude Code platform entry specifies the patterns

115 "CLAUDE.md", "claude.json", and ".claude/settings.json"; the MCP platform entry specifies ".mcp.json"; the Cursor platform entry specifies ".cursorrules" and ".cursor/rules/*.mdc"; the Windsurf platform entry specifies ".windsurfrules"; the Continue platform entry specifies ".continuerc.json" and "config.yaml"; the Aider platform entry specifies ".aider.conf.yml" and "aider.conf.yml"; and the Cline platform entry specifies

120 ".clinerules". The agent detection module (102) tests the received input path or paths against each platform's file patterns using case-insensitive glob matching. When a match is found, the corresponding platform identifier is returned along with a confidence score derived from the number of matching patterns. When no platform is recognized, the agent detection module (102) signals to the configuration parser (104) that generic parsing rules should be applied.

III. Configuration Parser (104)

125 The configuration parser (104) accepts the identified platform type from the agent detection module (102) and the raw file content, and produces a normalized in-memory representation of the configuration suitable for consumption by the capability extractor (106). The parser applies platform-specific parsing logic selected from the AI CLI registry (110) entry corresponding to the detected platform.

130 For JSON-formatted configuration files, the parser performs standard JSON deserialization followed by a recursive nested value extraction procedure. In a preferred embodiment, this procedure is implemented as a function findNestedValue(obj, key) that accepts an arbitrary JavaScript object and a target key string, traverses the object tree depth-first, and returns the first value associated with the specified key regardless of

135 nesting depth. This recursive approach is necessary because AI CLI configuration files frequently embed capability declarations at variable and non-standardized nesting depths. For example, the MCP server list in a .mcp.json file may be located at the top-level property "mcpServers", or may appear at a nested path such as "configuration.mcp.servers", depending on the tool version and user customization.

140 For YAML-formatted configuration files, the parser employs a YAML parsing library as a fallback deserialization strategy. The resulting parsed object is thereafter processed identically to a JSON-parsed object, enabling the capability extractor (106) to operate on a unified object representation regardless of the source serialization format. The YAML fallback also handles plain-text Markdown files by treating each non-empty line as a string token subject to pattern matching.

IV. Capability Extractor (106)

145 The capability extractor (106) receives the normalized configuration representation from the configuration parser (104) and produces a set of extracted capability declarations, each associated with a canonical capability type. In a preferred embodiment, the system recognizes eighteen canonical capability types: FILE_READ, FILE_WRITE, FILE_DELETE, CODE_EXECUTION, SHELL_ACCESS, NETWORK_ACCESS, BROWSER_AUTOMATION, PROCESS_MANAGEMENT,

150 MCP_TOOLS, API_ACCESS, GIT_OPERATIONS, ENV_VARIABLES, CREDENTIAL_ACCESS, SYSTEM_INFO, NOTIFICATION, DATA_EXFILTRATION, PROMPT_INJECTION, and PRIVILEGE_ESCALATION. These canonical types form the leaf-level and intermediate vocabulary of the capability taxonomy (112) depicted in FIG. 3.

155 The central operation of the capability extractor (106) is performed by a function mapToolToCapability(toolName), which accepts a string representing a tool name as declared in a configuration file and returns the corresponding canonical capability type. The mapping is implemented as a decision procedure that applies, in order: exact string matching against a lookup table of known tool names; prefix matching

160 against common tool-name prefixes such as "bash", "shell", "exec", "run", "file", "read", "write", "delete", "web", "browser", "git", "network", and "credential"; and, when neither exact nor prefix matching succeeds, a substring search for semantically significant terms within the tool name. For example, a tool named "mcp__playwright__browser_navigate" is mapped to BROWSER_AUTOMATION through prefix

165 matching on "browser", whereas a tool named "computer_use_tool" is mapped to SHELL_ACCESS through substring detection of "computer". A tool name not matched by any rule is classified as MCP_TOOLS, the default catch-all canonical type for unrecognized external tool invocations.

170 For Markdown-formatted configuration files such as CLAUDE.md, the capability extractor (106) applies a pattern-matching procedure against the textual content of the file. A library of regular-expression patterns corresponding to each canonical capability type detects phrases such as "allowed to execute bash commands" (matching SHELL_ACCESS), "can read and write files" (matching FILE_WRITE and FILE_READ), "may access the

175 internet" (matching NETWORK_ACCESS), and similar natural-language capability grants commonly found in AI agent instruction files. Each pattern match produces one capability declaration associated with the matched canonical type and annotated with the source file path and line number within the configuration file.

V. Risk Assessment Engine (108)

180 The risk assessment engine (108) accepts the set of capability declarations produced by the capability extractor (106) and the capability taxonomy (112), and produces a risk-annotated capability profile (114). For each capability declaration, the engine looks up the canonical capability type in the capability taxonomy (112) and retrieves the associated base risk level, which is one of CRITICAL, HIGH, MEDIUM, or LOW.

185 The assignment of base risk levels to canonical capability types is as follows. Capabilities in the CRITICAL tier (302) include FILE_DELETE (310), CODE_EXECUTION (312), SHELL_ACCESS (314), CREDENTIAL_ACCESS (316), DATA_EXFILTRATION, PROMPT_INJECTION, and PRIVILEGE_ESCALATION. Capabilities in the HIGH tier (304) include FILE_WRITE (320), NETWORK_ACCESS (322), BROWSER_AUTOMATION (324), and PROCESS_MANAGEMENT (326).

190 Capabilities in the MEDIUM tier (306) include MCP_TOOLS (330), API_ACCESS (332), GIT_OPERATIONS (334), and ENV_VARIABLES (336). Capabilities in the LOW tier (308) include FILE_READ (340), SYSTEM_INFO (342), and NOTIFICATION (344). This taxonomy structure is depicted graphically in FIG. 3, which shows the CAPABILITY TAXONOMY (300) root node branching to the four risk tier nodes and thence to their associated capability type leaf nodes.

195 The aggregate risk score is computed by the function calculateOverallRisk(capabilities), which implements a monotonically escalating combination rule. The function assigns numeric weights to each risk tier: CRITICAL maps to weight 4, HIGH to weight 3, MEDIUM to weight 2, and LOW to weight 1. The aggregate risk score is determined by the maximum tier weight present among all detected capabilities, escalated by a logarithmic factor proportional to

200 the count of capabilities at or above that tier. This escalation function ensures that a configuration file declaring many high-risk capabilities receives a higher aggregate score than one declaring a single high-risk capability, while preventing the aggregate score from being dominated by a large count of low-risk capabilities when higher-risk capabilities are also present.

VI. Report Generator (116)

205 The report generator (116) accepts the capability profile (114) produced by the risk assessment engine (108) and produces output artifacts in one or more of three formats: SARIF (Static Analysis Results Interchange Format), HTML, and JSON. The SARIF output conforms to the SARIF 2.1.0 specification and encodes each capability finding as a SARIF Result object, with the capability type as the rule identifier, the source file

210 path and line number as the location, and the risk level mapped to the SARIF severity enumeration (error for CRITICAL, warning for HIGH, note for MEDIUM, none for LOW). The HTML output provides a human-readable capability summary organized by risk tier, with a summary table showing total counts per tier and detailed per-capability findings including the source file, matched pattern, and remediation guidance.

215 The JSON output provides the complete capability profile as a machine-readable data structure suitable for ingestion by downstream security tooling, CI/CD pipeline gates, and policy enforcement systems. All three output formats include the aggregate risk score, the list of detected capabilities with their canonical types and base risk levels, and the source configuration file metadata.

VII. Method Description (FIG. 2)

220 Referring now to FIG. 2, the method of AI agent capability profile generation begins at START (202). At step 204, the system receives an AI CLI configuration file, which may be specified as a file path, a directory to scan recursively, or a set of file paths provided programmatically by a calling process. The system proceeds to step 206, where it applies the AI CLI registry (110) platform detection logic to identify the platform type associated

225 with the received file or files. At decision point 208, the system determines whether the detected file name and structure match any of the seven known platforms in the registry. If the platform is not recognized (NO branch from 208), the system proceeds to step 210, where generic parsing rules are applied. These rules treat the file as plain text if it lacks a recognized extension, as JSON if it has a .json extension, and as YAML if it has

230 a .yaml or .yml extension. After applying generic rules at step 210, or after a positive platform recognition (YES branch from 208), the method converges at step 212, where the configuration file is parsed according to the applicable parsing strategy.

235 At step 214, the parsed representation is analyzed to extract all capability declarations. For structured formats, this involves iterating over tool lists, permission arrays, and configuration keys. For Markdown-formatted files, this involves applying the regular expression pattern library. At step 216, each extracted declaration is mapped to its canonical capability type using the mapToolToCapability() function. At step 218,

240 each canonical capability type is assigned its base risk level from the capability taxonomy (112). At step 220, the aggregate risk score is computed using the calculateOverallRisk() function. At step 222, the complete capability profile (114) is assembled, comprising the list of capabilities, their risk levels, and the aggregate score. At step 224, the capability profile (114) is passed to the report generator

245 (116), which produces the output report in the requested format or formats. The method concludes at END (226).

VIII. Capability Taxonomy Structure (FIG. 3)

250 Referring now to FIG. 3, the capability taxonomy (300) is organized as a four-level hierarchy. The root node CAPABILITY TAXONOMY (300) branches to four risk tier nodes, each represented with a double-bordered box in the drawing to indicate its status as a classifier node rather than a leaf capability. The CRITICAL tier (302) subsumes FILE_DELETE (310), CODE_EXECUTION (312), SHELL_ACCESS (314), and CREDENTIAL_ACCESS (316).

255 The HIGH tier (304) subsumes FILE_WRITE (320), NETWORK_ACCESS (322), BROWSER_AUTOMATION (324), and PROCESS_MANAGEMENT (326). The MEDIUM tier (306) subsumes MCP_TOOLS (330), API_ACCESS (332), GIT_OPERATIONS (334), and ENV_VARIABLES (336). The LOW tier (308) subsumes FILE_READ (340), SYSTEM_INFO (342), and NOTIFICATION (344).

260 The taxonomy is designed to be extensible; additional capability types may be introduced as leaf nodes under any existing tier node without modification of the risk assessment engine (108). Similarly, additional tier nodes may be introduced between existing tiers to accommodate future refinements in risk classification granularity.

Abstract

335 A system and method automatically generate a structured AI agent capability profile from configuration files associated with AI coding assistant CLI tools. An agent detection module identifies the AI CLI platform type by matching file names against a registry of known platforms including Claude Code, Cursor, Windsurf, Continue, Aider, Cline, and MCP. A configuration parser deserializes the configuration file using JSON parsing with YAML fallback, applying recursive nested value extraction. A capability extractor maps tool names and permission declarations to canonical capability types using a three-stage matching procedure. A risk assessment engine assigns base risk levels from a four-tier capability taxonomy comprising CRITICAL, HIGH, MEDIUM, and LOW tiers, and computes an aggregate risk score using a monotonically escalating combination function. A report generator produces output in SARIF, HTML, and JSON formats. The method operates entirely through static analysis without executing any AI CLI tooling.
