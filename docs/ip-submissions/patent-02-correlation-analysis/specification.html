Technical Field

15 The present invention relates generally to multi-file static analysis and artificial intelligence security, and more particularly to systems and methods that detect coordinated attack chains distributed across multiple AI CLI configuration files by constructing a file relationship graph, searching for cross-file content patterns, and computing a quantitative correlation strength score that determines whether a multi-file finding should be emitted.

20 More specifically, the invention concerns the exploitation of structural relationships unique to AI CLI configuration environments — namely directory proximity between files and semantic naming conventions governing subdirectory types such as hooks, skills, and agents — as a basis for inferring whether multiple independently benign configuration artifacts form a coherent attack chain when considered in combination.

Background of the Invention

25 AI CLI configuration environments present a novel and previously unaddressed attack surface for static security analysis. A sophisticated attacker who has obtained write access to a developer's project directory can distribute a coordinated attack chain across multiple configuration files such that no individual file, when analyzed in isolation, exhibits characteristics sufficient to trigger conventional security alerts.

30 For example, an attacker may place a data collection script at a path such as hooks/post-commit.sh, which is a standard location for post-commit hook scripts in Claude Code environments, and separately place an exfiltration instruction in a skills file at skills/exfil.md, which superficially resembles a legitimate agent skill definition. Neither file, in isolation, would necessarily raise a red flag under

35 conventional pattern matching, because the hook file contains only standard shell constructs and the skills file contains only natural-language text. The attack is revealed only when the two files are analyzed together and their combined semantic content is understood to constitute a coordinated credential harvesting and exfiltration chain.

40 Traditional cross-file taint analysis techniques rely on program flow graphs, call graphs, or data flow graphs extracted from source code. These techniques require that the files under analysis contain formal programming constructs — function declarations, variable assignments, control flow statements — that can be parsed into an abstract syntax tree and connected across files through resolved import, require, or include directives.

45 AI CLI configuration files do not satisfy these requirements. They are predominantly comprised of Markdown-formatted natural language, YAML or JSON-serialized configuration objects, and shell scripts that are never directly imported by or linked to other files in the configuration environment. Consequently, traditional cross-file taint analysis does not apply to this domain, and no prior art system has addressed the challenge of detecting coordinated attack chains distributed across AI CLI configuration files.

50 The AI CLI configuration environment, however, exhibits two structural properties that are unique to this domain and that serve as a substitute for the program-flow connectivity exploited by traditional cross-file taint analysis. First, configuration files that are relevant to the same AI agent context are almost always stored in the same project directory or in closely related subdirectories, making directory proximity a reliable indicator of file relationship. Second, AI CLI platforms impose a conventional directory

55 structure in which specific subdirectory names such as "hooks", "skills", and "agents" denote files that interact at runtime: hook files are executed by the AI platform at defined lifecycle points, skill files define behavioral capabilities that agents can invoke, and agent files define autonomous sub-agents. Files residing in these conventionally related subdirectories are therefore related by semantic naming convention even if no explicit import or dependency declaration links them.

60 There is therefore a need for a system and method that constructs a file relationship graph encoding both directory proximity and semantic naming relationships, applies multi-file content pattern matching rules against sets of related files, computes a quantitative correlation strength score, and emits structured findings for file combinations whose correlation strength meets or exceeds a configurable threshold.

Summary of the Invention

65 The present invention provides a system and method for detecting coordinated attack chains distributed across multiple AI CLI configuration files. The system constructs a file relationship graph in which nodes represent configuration files and edges represent relationships established through directory proximity or semantic naming conventions. The system applies a library of correlation rules, each specifying file-type patterns and

70 content patterns that, when co-occurring across related files, indicate a potential attack chain. For each correlation rule, the system searches for pattern matches across the related file sets identified by the graph, computes a quantitative correlation strength score using a formula that rewards both content pattern coverage and cross-file distribution of matches, and emits a structured correlation finding when the score meets or exceeds a configurable threshold.

75 In a preferred embodiment, the system infers one or more risk vectors from each emitted correlation finding, classifying the finding according to the type of attack it represents, such as Credential Exposure, Data Exfiltration, Command Injection, Persistent Backdoor, or Supply Chain Compromise. The complete correlation finding, comprising the related files, correlation strength score, and inferred risk vectors, is provided as output for consumption by downstream security tooling or human review.

Detailed Description of the Preferred Embodiments

100 Reference will now be made in detail to the presently preferred embodiments of the invention, examples of which are illustrated in the accompanying drawings. Wherever possible, the same reference numbers will be used throughout the drawings to refer to the same or like parts.

I. File Relationship Graph Construction (FIG. 1)

105 Referring now to FIG. 1, the file relationship graph is a data structure in which each node represents a configuration file discovered in the target directory tree, and each edge represents a relationship between two files. Three types of edges are recognized. A directory proximity edge (212) is drawn between two files when the number of path components separating their respective containing directories does not exceed a threshold,

110 which in a preferred embodiment is set to two. This threshold is justified by the observation that AI CLI configuration files relevant to the same agent context are consistently stored within two directory levels of one another in standard project layouts. A semantic naming edge (214) is drawn between two files when their containing directories are members of a predefined set of semantically related directory-name pairs. A dual-type

115 edge (216) is drawn when both directory proximity and semantic naming criteria are satisfied simultaneously. In FIG. 1, the bold solid line between hooks/post.sh (206) and agents/helper.md (210) represents such a dual-type edge.

120 The graph is constructed by a function buildFileRelationships(files) that accepts a list of file path strings and returns a map associating each file path with a set of related file paths. The function computes the directory distance between every ordered pair of files and, for pairs whose distance does not exceed the proximity threshold, adds a directory proximity edge. For every pair of files, the function also

125 invokes a subsidiary function areFilesRelatedByNaming(fileA, fileB), which extracts the last path component of the containing directory for each file and tests whether that pair of directory names appears in the predefined semantic naming pattern table. If so, a semantic naming edge is added for that pair.

130 The directory distance metric is computed as the count of path components that must be traversed to reach one directory from the other within the file system tree. For two files located at paths /project/.claude/hooks/post.sh and /project/.claude/skills/exfil.md, the respective containing directories are /project/.claude/hooks and /project/.claude/skills. These directories share the common ancestor /project/.claude,

135 and each directory requires one step to reach that ancestor, yielding a total directory distance of two. This distance is equal to the proximity threshold, so these two files receive a directory proximity edge.

II. Semantic Naming Patterns

140 The semantic naming pattern table maintained by the areFilesRelatedByNaming() function comprises a plurality of ordered and unordered directory-name pairs that represent known runtime interaction relationships in AI CLI configuration environments. In a preferred embodiment, the semantic naming pattern table includes the following pairs: hooks ↔ skills, indicating that hook execution scripts and skill definition files interact through the

145 agent's skill invocation mechanism; hooks ↔ agents, indicating that hook execution scripts and agent definition files interact through the agent lifecycle management mechanism; CLAUDE.md ↔ .mcp.json, indicating that the primary behavioral instruction file and the MCP server declaration file are consumed by the same agent process and therefore have combined semantic effect; settings.json ↔ config.*, indicating that settings files and

150 configuration files with variable suffixes are frequently used in conjunction to define the same agent configuration; and agent ↔ skill, indicating that agent definition files and skill definition files interact through skill invocation.

155 The semantic naming pattern matching is case-insensitive and applies to the terminal directory name component. For files stored directly in the project root without a distinguishing subdirectory, the containing directory name is the project root name, and semantic naming relationships are evaluated against the file names themselves rather than their containing directory names in this special case.

III. Correlation Rules

160 The correlation analysis system maintains a library of correlation rules, each of which specifies the conditions under which a multi-file attack chain should be flagged. Each correlation rule comprises three fields: a filePatterns array, a contentPatterns array, and a description string. The filePatterns array specifies glob patterns matching the types of files that

165 are expected to participate in this type of attack chain. For example, a rule targeting hook-and-skill coordination attacks might specify the file patterns ["**/*.sh", "**/*.md"] to match shell scripts and Markdown files. The contentPatterns array specifies regular expression patterns that, when matched across the participating files, indicate the presence of malicious content or coordinated intent. The

170 description field provides a human-readable characterization of the attack chain type for use in generated findings, such as "Hook execution with exfiltration skill coordination" or "MCP tool injection with credential harvesting."

175 The correlation rule library is extensible without modification of the analysis engine. New rules may be added by appending entries to the rule array, and each entry is processed uniformly by the cross-file pattern matching and correlation strength computation logic. This design ensures that the invention remains applicable to novel attack patterns discovered after the system's initial deployment.

IV. Cross-File Pattern Matching

180 For each correlation rule, the system executes a function findCrossFilePatterns(rule, files, relationships) that accepts the correlation rule, the full set of discovered configuration files, and the file relationship graph, and returns a cross-file match result comprising the set of files in which at least one content pattern match was found and the set of content patterns that produced at least one match across the file set.

185 The function first identifies the candidate file set for the rule by filtering the full file list to retain only files whose paths match at least one of the rule's filePatterns glob patterns. It then groups the candidate files into relationship clusters by consulting the file relationship graph: two files are placed in the same cluster if they are connected by at least one edge in the graph.

190 For each relationship cluster, the function reads the content of each file in the cluster and applies each regular expression in the rule's contentPatterns array against the file content. A content pattern is considered matched for the cluster if it produces at least one match in any file within the cluster. The function records which files contributed at least one match and which patterns were matched, producing the cross-file match result.

V. Correlation Strength Computation (FIG. 2)

195 Referring now to FIG. 2, the correlation strength score S for a given cluster and correlation rule is computed according to the formula:

S = (M / P) + min(F / Fₙ, 1) × 0.2 + 0.1

200 where M (220) is the number of content patterns in the rule that were matched across the file cluster; P (222) is the total number of content patterns specified in the rule; F (224) is the count of distinct files in the cluster that contributed at least one pattern match; and Fₙ (226) is the count of file patterns specified in the rule. The resulting score S is capped at 1.0 (228).

205 The formula has three additive components. The primary component M/P measures content coverage: the fraction of the rule's content patterns that were actually matched. A cluster that matches all of a rule's content patterns achieves a primary component value of 1.0, while a cluster that matches none achieves 0.0. The secondary component min(F/Fₙ, 1) × 0.2 measures file distribution: the degree to which the matched

210 content is spread across multiple files as expected by the rule, rather than concentrated in a single file. This component rewards cross-file distribution of evidence and penalizes cases where all matched content appears in a single file, which is less indicative of a coordinated multi-file attack chain. The constant additive term 0.1 represents a base presence bonus awarded whenever any file in the cluster matches any

215 file pattern in the rule, ensuring that a cluster scoring 0 on both primary and secondary components but satisfying the file type requirements receives a non-zero score.

220 The threshold bar illustrated at the bottom of FIG. 2 represents the configurable emit-finding threshold (230). The system emits a correlation finding for a cluster only when its computed score S meets or exceeds this threshold. In a preferred embodiment, the default threshold value is 0.3, which is positioned at approximately 30% of the 0.0-to-1.0 score range. This default is calibrated to detect attack chains in which at

225 least one content pattern is matched and the matched content appears in more than one file, while avoiding false positives from random co-occurrence of file types without meaningful content pattern coverage.

VI. Risk Vector Inference

230 For each correlation finding whose score S meets or exceeds the threshold, the system invokes a function generateRiskVectors(finding) that accepts the finding and returns an array of risk vector strings. The function derives the risk vectors by analyzing the human-readable description field of the matched correlation rule using a secondary pattern-matching step that identifies semantically significant phrases.

235 Specifically, the function detects the presence of terms associated with each of the following recognized risk vector categories: Credential Exposure, detected by phrases such as "credential", "token", "key", "secret", "password", "api key", or "auth"; Data Exfiltration, detected by phrases such as "exfil", "exfiltrat", "leak", "send data", or "transmit"; Command Injection, detected by phrases such as "inject",

240 "command injection", "shell inject", or "code inject"; Persistent Backdoor, detected by phrases such as "backdoor", "persist", "startup", "autorun", or "cron"; and Supply Chain Compromise, detected by phrases such as "supply chain", "dependency", "package", "registry", or "upstream". A single correlation finding may be assigned multiple risk vectors when its description contains terms matching more than one category.

245 The CorrelationFinding data structure produced as output by the system comprises the following fields: relatedFiles, an array of the file paths in the matched cluster; correlationStrength, the numeric score S computed for the cluster; riskVectors, the array of risk vector strings produced by generateRiskVectors(); rule, the correlation rule that

250 produced the finding; and matchedPatterns, the subset of the rule's content patterns that produced at least one match across the related files. This structure provides sufficient information for downstream security tooling to categorize, prioritize, and remediate the identified attack chain.

VII. Analysis Pipeline Method (FIG. 3)

255 Referring now to FIG. 3, the analysis pipeline method begins at START (232). At step 234, the system discovers configuration files in the target directory by scanning recursively for files whose names or paths match any of the known AI CLI configuration file patterns. The discovery step returns a list of file paths for further analysis.

260 At step 236, the system constructs the file relationship graph from the discovered files using the buildFileRelationships() function. This step uses two side inputs: the directory distance calculation logic (238), which computes the number of path components between file directories, and the semantic naming patterns table (240), which specifies the recognized directory-name pairs. The output of step 236 is a complete

265 file relationship graph encoding both proximity and naming edges between all relevant file pairs.

270 At step 242, the system iterates over each correlation rule in the rule library. For each rule, at step 244, the system invokes findCrossFilePatterns() to identify which related file clusters contain matches for the rule's file patterns and content patterns. At step 246, the correlation strength score S is computed for each cluster using the formula of FIG. 2.

275 At decision point 248, the system determines whether the computed score S meets or exceeds the configured threshold. If the threshold is not met (NO branch from 248), the system returns to step 242 to process the next rule, without emitting any finding for the current cluster and rule combination. If the threshold is met (YES branch from 248), the system proceeds to step 250, where generateRiskVectors() is invoked to infer the

280 risk vectors associated with the finding. At step 252, a complete CorrelationFinding structure is emitted as output, comprising the related files, correlation strength score, matched patterns, and inferred risk vectors. After all rules have been processed for all clusters, the method concludes at END (254).

VIII. Configurability and Extension

285 The emit-finding threshold (230) is configurable at runtime, enabling security teams to adjust sensitivity to match their operational requirements. A lower threshold value increases recall at the cost of additional false positives; a higher threshold value increases precision at the cost of potentially missing lower-confidence attack chains. In a preferred embodiment, the threshold is specified as a floating-point number in the

290 range [0.0, 1.0] and is passed to the analysis function as a configuration parameter with a default value of 0.3. The threshold may be specified globally, applying uniformly to all correlation rules, or may be specified per-rule, enabling fine-grained tuning of detection sensitivity for specific attack chain types.

295 The semantic naming pattern table is also configurable, enabling the invention to be extended to support novel AI CLI platforms that employ different subdirectory naming conventions without modification of the core graph construction or correlation analysis logic. New platform-specific naming patterns are added by appending entries to the pattern table, and take effect immediately upon the next analysis run.

Abstract

360 A system and method detect coordinated attack chains distributed across multiple AI CLI configuration files by constructing a file relationship graph whose edges encode directory proximity and semantic naming relationships unique to AI CLI environments, including hooks-to-skills, hooks-to-agents, and CLAUDE.md-to-.mcp.json conventions. For each correlation rule specifying file-type patterns and content patterns, the system searches for cross-file pattern matches within related file clusters and computes a correlation strength score using the formula S = (M/P) + min(F/Fₙ, 1) × 0.2 + 0.1. Findings are emitted for clusters whose score meets or exceeds a configurable threshold. Each finding is annotated with inferred risk vectors — including Credential Exposure, Data Exfiltration, and Command Injection — derived from secondary pattern matching on the rule description. The method operates entirely through static analysis without executing any AI CLI tooling.
