I. BACKGROUND AND PROBLEM STATEMENT

Modern AI coding assistants (Claude Code, Cursor, Windsurf, Continue, Aider, Cline) are configured through markdown-formatted files such as CLAUDE.md, .cursorrules, and AGENTS.md. These files occupy a unique position in software security: they contain natural-language instructions to AI systems, but they also routinely embed fenced code blocks, shell scripts, and executable snippets intended to demonstrate desired behaviors or configure automated workflows.

Prior art in static analysis focuses on analyzing actual source code files — JavaScript, TypeScript, Python, etc. — where the primary artifact is executable code. No prior tool analyzes markdown configuration files as a mixed-type artifact where natural language sections and embedded code blocks must be treated differently and where the security implications arise from AI interpretation rather than direct execution.

The problem addressed by this invention: embedded code blocks in AI configuration markdown can instruct AI agents to perform dangerous operations. A prompt-injection attacker can embed a fenced code block containing a shell command, dynamic import chain, or dangerous function call within what appears to be documentation. The AI agent, upon reading its configuration file, may execute these instructions literally. No existing security scanner extracts, parses, and analyzes these embedded code blocks using formal Abstract Syntax Tree (AST) analysis as distinct artifacts from the surrounding prose.

II. SUMMARY OF THE INVENTION

The present invention provides a method and system for extracting fenced code blocks from markdown-formatted AI configuration files, submitting each extracted block to language-appropriate AST parsing, and applying a catalog of semantic security patterns to the resulting AST to identify dangerous constructs — all without executing any code. The invention addresses threats including dynamic module loading, dangerous built-in function invocations, property-chain traversals to security-sensitive objects, and object structures designed to exfiltrate or relay data.

A key novelty is the semantic pattern matching layer: rather than simple string or regex matching against raw text, the invention matches against AST node structures, providing resistance to obfuscation techniques such as Unicode escaping, string concatenation, and identifier aliasing.

III. DETAILED DESCRIPTION OF THE INVENTION

A. System Architecture

Figure 1. Five-stage analysis pipeline: markdown input → code block extraction → AST parsing → semantic pattern matching → risk findings.

B. Code Block Extraction (Step 1)

The extractCodeBlocks() function implements a two-pass state machine over the lines of a markdown file. In the first pass, it identifies fenced code blocks delimited by triple-backtick sequences including optional language tags (e.g., ```javascript, ```bash). Each extracted block is represented as an object with the following properties:

  • language: The language hint from the opening fence (normalized to lowercase)
  • content: The verbatim text content between the fences
  • startLine / endLine: Source line numbers within the markdown file
  • context: Up to 3 lines of prose immediately preceding the block

The second pass applies language detection heuristics to blocks with absent or ambiguous language tags, using token frequency analysis to distinguish JavaScript/TypeScript from shell scripts, Python, and other languages. This is critical because AI configuration files frequently omit language tags.

C. AST Construction (Step 2)

Each extracted code block with a recognized scripting language (JavaScript, TypeScript) is submitted to the TypeScript compiler API (ts.createSourceFile()) to produce a full Abstract Syntax Tree. The AST captures structural relationships between tokens — function calls, property chains, import expressions, and object literals — rather than treating the code as flat text. The use of a compiler-grade AST provides resistance to trivial obfuscation: whitespace insertion, line breaks, and comment injection do not affect AST structure.

Shell script blocks are processed by a purpose-built command tokenizer rather than a full AST parser, which identifies dangerous command patterns (data exfiltration utilities, remote execution, process substitution) in a structure-aware manner.

D. Semantic Pattern Catalog (Step 3)

Figure 2. Taxonomy of five semantic security pattern types applied to extracted ASTs, with examples of each category.

E. Semantic Pattern Matching Algorithm

The matchSemanticPattern() function performs AST traversal using recursive descent. For each node in the AST, it evaluates the node against each registered pattern based on the pattern's type discriminant:

function-call patterns: Match AST call expression nodes (SyntaxKind.CallExpression) where the callee identifier or member expression path appears in a catalog of dangerous function names. The catalog includes process termination, child process spawning, network socket creation, and indirect code execution functions. Matching is performed on the resolved callee name after stripping wrapper parentheses and type assertions.

property-access patterns: Match AST property access expression nodes (SyntaxKind.PropertyAccessExpression) where the object.property chain traverses known security-sensitive paths. The invention checks multi-level chains (e.g., object.child.sensitive) by following parent-child relationships in the AST, enabling detection of deeply nested accesses.

dynamic-import patterns: Match AST dynamic import expressions (SyntaxKind.ImportExpression, SyntaxKind.CallExpression with callee identifier "require") where the module specifier is not a string literal. Variable, template literal, and concatenation-expression module specifiers are flagged, as they indicate runtime-determined module loading which cannot be statically validated.

indirect-exec patterns: Match AST call expressions that convert strings into executable code at runtime. This category targets the "eval-chain" threat: call expressions whose callee resolves to well-known dynamic execution identifiers, Function constructor instantiation, and timer functions invoked with string arguments rather than function references. These are primary mechanisms for obfuscated prompt injection attacks.

object-structure patterns: Match AST object literal expressions (SyntaxKind.ObjectLiteralExpression) whose property key sets match the signature of known dangerous data transfer objects: fetch configuration objects (url + method + headers + body), API credential containers, and webhook payload objects. This detects code that constructs data exfiltration payloads even when the actual transmission function call is in a separate block.

F. Cross-Block Correlation

A significant novel aspect of the invention is cross-block analysis: findings from multiple code blocks within the same markdown file are correlated to identify split attack chains. For example, an object-structure finding in block #3 (constructing an exfiltration payload) combined with a property-access finding in block #7 (reading environment variables) produces a composite "credential exfiltration chain" finding of elevated severity, even though neither block alone would warrant a critical alert. This correlation is implemented in the analyzeFile() method by maintaining a finding accumulator across all blocks and applying combination rules after all individual-block analyses complete.

IV. INDEPENDENT CLAIMS

Claim 1 (Method — Independent): A computer-implemented method for analyzing security threats in AI agent configuration files, comprising: receiving a markdown-formatted configuration file associated with an AI coding assistant; extracting one or more fenced code blocks from the markdown content, each block including a content string and source position metadata; for each extracted code block having a recognized scripting language, constructing an Abstract Syntax Tree from the code block content using a compiler-grade parser; traversing the Abstract Syntax Tree to identify AST nodes matching one or more semantic security patterns selected from the group consisting of: dangerous function call patterns, security-sensitive property access chain patterns, dynamic module loading patterns, indirect code execution patterns, and data exfiltration object structure patterns; generating a security finding for each matched pattern, including the source position within the original markdown file; and correlating findings across multiple code blocks within the same file to identify composite attack chains.
Claim 2 (System — Independent): A system for detecting security threats in AI coding assistant configuration files, comprising: one or more processors; memory storing instructions that, when executed, implement: a code block extraction module configured to parse markdown content and extract fenced code blocks with associated language tags and source line numbers; an AST construction module configured to invoke a TypeScript compiler API to generate an Abstract Syntax Tree from each extracted code block; a semantic pattern matching module configured to traverse the Abstract Syntax Tree and match AST node structures against a catalog of security patterns using node type and identifier-based matching; a cross-block correlation module configured to identify composite attack chains by combining findings from multiple code blocks in the same file; and a finding reporting module configured to output security alerts with source positions traceable to the originating markdown file.
Claim 3 (Dependent on Claim 1): The method of claim 1, wherein the indirect code execution patterns match call expressions where the callee identifier resolves to a dynamic code evaluation function, including function constructor instantiation, and timer functions invoked with a string argument as the first parameter rather than a function reference.
Claim 4 (Dependent on Claim 1): The method of claim 1, wherein extracting fenced code blocks further comprises applying language detection heuristics to code blocks that lack language tags, the heuristics comprising token frequency analysis to classify the block as JavaScript/TypeScript, shell script, or other language.
Claim 5 (Dependent on Claim 1): The method of claim 1, wherein correlating findings further comprises applying severity escalation rules such that a composite finding derived from combining findings of two or more pattern types is assigned a severity level higher than either individual finding.
Claim 6 (Dependent on Claim 1): The method of claim 1, wherein the markdown-formatted configuration file is one of: a CLAUDE.md file associated with the Claude Code AI coding assistant; a .cursorrules file associated with the Cursor AI coding assistant; an AGENTS.md file associated with a multi-agent AI system; or a system prompt configuration file associated with any large language model agent.
