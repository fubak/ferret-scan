Technical Field

15 [0002] The present invention relates generally to static program analysis and software security, and more particularly to systems and methods for parsing and analyzing fenced code blocks embedded within markdown-formatted configuration files used by artificial intelligence coding assistant command-line interface (CLI) tools, by constructing and traversing Abstract Syntax Trees (ASTs) derived from said code blocks so as to identify semantic security patterns indicative of malicious, dangerous, or policy-violating behavior. The invention further relates to compiler technology, language-theoretic security analysis, and enterprise governance of AI-assisted software development environments.

Background of the Invention

25 [0003] AI coding assistants have become integral to modern software development workflows. Platforms such as Claude Code (Anthropic), Cursor, Windsurf, Aider, Continue, and Cline permit developers to delegate file manipulation, shell execution, network access, and third-party tool invocation to AI agents operating within local or cloud-connected development environments. These platforms are typically configured through one or more markdown-formatted files stored at the project root or in the user home directory.

35 [0004] Examples of such configuration files include CLAUDE.md (Claude Code, Anthropic), .cursorrules (Cursor, Anysphere), AGENTS.md (OpenAI Codex CLI), and .windsurfrules (Windsurf, Codeium). A distinctive and previously unanalyzed property of these configuration files is that they are mixed-type artifacts: they interleave natural-language prose directives with fenced code blocks delimited by triple backticks (```) or tildes. These embedded code blocks may contain JavaScript, TypeScript, Python, Shell, or other executable content that an AI agent may interpret and execute when processing the configuration file.

45 [0005] Adversaries or malicious insiders may therefore embed harmful executable instructions within ostensibly benign markdown configuration files, with the expectation that the AI agent will execute the embedded code, exfiltrate credentials, invoke unauthorized external processes, or otherwise compromise the security of the development environment. Because these configuration files are frequently committed to version control repositories and shared across development teams, a single compromised configuration file can silently propagate a harmful code payload to every developer who clones or pulls the repository.

55 [0006] Prior art static analysis tools (SAST) are designed to operate on source code files with well-defined syntactic structure; none of the prior art treats markdown configuration files as the mixed-type artifacts they are, requiring separate extraction and independent analysis of each embedded code block. Prior art tools that do attempt to scan configuration files do so through string matching or regular expression techniques applied to the raw text of the file. Such approaches are fundamentally obfuscation-susceptible: an adversary may introduce whitespace, comment nodes, aliased variable references, or syntactic transformations that preserve the operational semantics of a dangerous pattern while defeating any string or regex signature.

65 [0007] There is therefore a need for a system and method that extracts each fenced code block from a markdown configuration file, constructs a true Abstract Syntax Tree from the extracted block content, and traverses that AST to match semantic security patterns that are invariant to obfuscation-class transformations including whitespace variation, comment insertion, string literal splitting, and identifier aliasing. No prior art system has addressed this specific combination of mixed-type document parsing and AST-based semantic security pattern matching applied to AI agent configuration files.

Summary of the Invention

75 [0008] The present invention provides a computer-implemented system and method for analyzing fenced code blocks embedded in markdown configuration files used by AI coding assistant CLI platforms. A code block extractor (302) processes the markdown configuration file (300) using a state-machine parser that identifies triple-backtick or tilde-delimited fenced regions, capturing for each block its language identifier tag, raw content, start line number, end line number, and contextual prose appearing immediately before the block. Each extracted block is represented as a block object (316) and subjected to language detection (304) to resolve ambiguous or untagged blocks via token-frequency heuristics. An AST parser (306) then constructs a structural Abstract Syntax Tree from the block content using an appropriate language compiler frontend.

87 [0009] A semantic pattern matcher (308) traverses the AST using recursive descent to identify nodes that match one or more security-relevant semantic patterns drawn from a pattern catalog (310), including function-call patterns (322), property-access patterns (324), dynamic-import patterns (326), indirect-execution patterns (328), and object-structure patterns (329). Findings are emitted by a finding emitter (314) and subjected to cross-block correlation (312) to detect composite attack chains that span multiple code blocks within the same configuration file. Because the analysis operates on the AST structural representation rather than on raw text, the detection is resistant to a broad class of obfuscation techniques including whitespace variation, comment insertion, variable aliasing, and syntactic reformatting.

99 [0010] In a second aspect, the invention provides a method comprising: receiving a markdown configuration file associated with an AI CLI platform; extracting from said file one or more fenced code blocks together with source position metadata; applying language detection heuristics to blocks lacking explicit language identifiers; constructing an Abstract Syntax Tree from each extracted block; traversing said AST by recursive descent to identify AST nodes matching semantic security patterns; recording each matching node as a finding annotated with its source file position; applying cross-block correlation rules to a set of findings accumulated from all blocks in the file; and emitting a set of composite findings representing potential security violations or attack chains present in the markdown configuration file.

I. System Architecture Overview (FIG. 1)

162 [0016] Referring now to FIG. 1, the system for security analysis of embedded code blocks comprises a pipeline originating at a markdown configuration file input (300). The markdown configuration file (300) is a text file in Markdown format, as commonly used by AI CLI platforms to convey behavioral instructions, tool allowances, and contextual guidance to an AI coding assistant agent. Examples include CLAUDE.md (Claude Code, Anthropic), .cursorrules (Cursor, Anysphere), and AGENTS.md (OpenAI Codex CLI). These files may reside at the project root directory, the user home directory, or any directory in the file system hierarchy that the AI agent traverses during its initialization sequence.

174 [0017] The pipeline components are, in order of execution: code block extractor (302), language detector (304), AST parser (306), semantic pattern matcher (308), and finding emitter (314). An auxiliary pattern catalog (310) provides semantic pattern definitions to the semantic pattern matcher (308). A cross-block correlator (312) operates as a post-processing stage that receives the accumulated finding set from the finding emitter (314) and applies combination rules across findings from distinct code blocks to produce composite findings representing multi-stage attack chains.

II. Code Block Extractor (302)

184 [0018] The code block extractor (302) receives the markdown configuration file (300) and produces a sequence of extracted block objects (316). In a preferred embodiment, the extractor (302) is implemented as a finite-state machine with the following states: PROSE (initial state), FENCE_OPEN (a triple-backtick or tilde delimiter has been encountered), BLOCK_CONTENT (currently accumulating code block content), and FENCE_CLOSE (a closing delimiter has been matched). Upon transitioning from FENCE_CLOSE back to PROSE, the extractor commits a completed extracted block object (316) to its output sequence.

196 [0019] Each extracted block object (316) comprises the following fields: (a) language, the language identifier tag appearing after the opening backtick delimiter, if present; (b) content, the raw string content of the fenced block, exclusive of the delimiter lines; (c) startLine, the one-indexed line number of the opening delimiter within the source markdown file; (d) endLine, the one-indexed line number of the closing delimiter within the source markdown file; and (e) context, a string comprising up to three lines of prose text immediately preceding the opening delimiter, which serves as a semantic label describing the intended purpose of the code block as expressed by the file's author. This context field enables downstream components to correlate AST-derived findings with the author-supplied description of the code block's purpose, which may itself be relevant to risk assessment.

III. Language Detector (304)

210 [0020] The language detector (304) resolves the language of each extracted block object (316). When the language field is populated with a recognized language identifier such as "javascript", "typescript", "python", "bash", or "sh", the language detector (304) passes the tag through directly to the AST parser (306). When the language field is absent or contains an unrecognized token, the language detector (304) applies a set of language heuristics to infer the most probable language of the block content.

220 [0021] In a preferred embodiment, these heuristics comprise token-frequency analysis: the block content is tokenized by whitespace and punctuation, and the resulting token set is compared against language-specific signature token sets. For example, the presence of def and import tokens at line beginnings is predictive of Python; the presence of const, let, var, =>, and semicolon tokens is predictive of JavaScript or TypeScript; the presence of shebang sequences (#!/) or shell builtins such as export, source, and chmod is predictive of shell script content. The heuristic assigns a confidence score to each candidate language and selects the language with the highest confidence score. If no language achieves a confidence score above a configurable threshold, the block is marked as language-unknown and is skipped by the AST parser (306) in the current embodiment, though alternate embodiments may apply generic or polyglot parsing strategies to unknown-language blocks.

IV. AST Parser (306)

236 [0022] The AST parser (306) receives each extracted block object (316) together with its resolved language identifier and constructs a structured Abstract Syntax Tree representation of the block content. In a preferred embodiment for JavaScript and TypeScript blocks, the AST parser (306) invokes the TypeScript compiler API function ts.createSourceFile(), passing the block content string as input and configuring the compiler frontend to tolerate syntactic errors so that partial or informally written code blocks — as commonly appear in configuration file examples — are parsed to the greatest degree possible rather than rejected entirely.

247 [0023] The resulting TypeScript compiler AST is a fully typed tree structure in which each node carries a kind discriminant drawn from the ts.SyntaxKind enumeration, as well as positional metadata (pos and end character offsets within the source text) that the finding emitter (314) translates back to line and column numbers for reporting purposes. For Python blocks, the AST parser (306) invokes the Python ast.parse() standard library function, obtaining a Python AST in which each node carries a type attribute and line/column annotations. For shell script blocks, the AST parser (306) invokes a shell grammar parser to obtain a structured command-word tree. Alternate embodiments may employ any language-appropriate parser frontend that produces a structured, traversable AST representation with positional annotations, including third-party parsers such as Babel, Acorn, Esprima, Tree-sitter, or ANTLR-generated parsers for languages not natively supported by the TypeScript compiler API.

V. Semantic Pattern Catalog (310, 320, FIG. 2)

263 [0024] Referring now to FIG. 2, the pattern catalog (310) organizes security-relevant semantic patterns under the root node SEMANTIC PATTERN CATALOG (320). The catalog defines five pattern type branches, each associated with a specific AST node class and a corresponding matching algorithm. These five pattern types collectively represent the principal mechanisms by which executable code embedded in markdown configuration files may perform security-violating operations. Each pattern type is described in detail below.

272 [0025] A. Function-Call Pattern (322). The function-call pattern (322) targets CallExpression AST nodes in which the callee expression resolves to a function identifier or member expression present in a configurable catalog of security-sensitive function names. In a TypeScript/JavaScript AST, a CallExpression node has a callee child node and an arguments child list. The pattern matcher (308) traverses the callee expression recursively to resolve the fully qualified function name, accounting for member access chains. The resolved name is then tested against a dangerous-function catalog maintained in the pattern catalog (310). The catalog includes entries for process management functions, file system access functions, network socket operations, cryptographic operations with weak parameters, and other functions whose invocation in a configuration-file code block indicates a potential policy violation. Because this matching operates on the CallExpression node structure rather than on the raw text of the source, aliased calls are detected at the AST level through data-flow-sensitive variants of the matcher, and obfuscating whitespace or comment nodes within the call expression have no effect on the structural match.

289 [0026] B. Property-Access Pattern (324). The property-access pattern (324) targets PropertyAccessExpression AST nodes representing member access chains on security-sensitive objects. In JavaScript and TypeScript, chained property accesses such as process.env.SECRET_KEY or os.environ["HOME"] produce a left-recursive PropertyAccessExpression tree. The pattern matcher (308) traverses such chains by iteratively unwrapping PropertyAccessExpression and ElementAccessExpression nodes to reconstruct the full dotted access path. The reconstructed path is then tested against a catalog of security-sensitive access paths, including environment variable access paths, process namespace paths, file system descriptor paths, and network interface paths. Detection of such paths in a configuration-file code block indicates potential credential exfiltration, capability probing, or system information gathering.

302 [0027] C. Dynamic-Import Pattern (326). The dynamic-import pattern (326) targets ImportExpression AST nodes in which the module specifier argument is not a string literal. In ECMAScript 2020 and later, the import() expression enables runtime module loading, and a dynamic (non-literal) specifier argument indicates that the module path is computed at runtime — potentially from user-controlled data or from the configuration file's own embedded variables. Such dynamic imports in a configuration-file code block represent a potential mechanism for loading arbitrary or attacker-controlled modules at AI agent initialization time. The pattern matcher (308) identifies ImportExpression nodes whose specifier child is not a StringLiteral or a TemplateLiteral with no substitutions, flagging all other specifier types as matching the dynamic-import security pattern (326).

316 [0028] D. Indirect-Execution Pattern (328). The indirect-execution pattern (328) targets CallExpression and NewExpression AST nodes that perform string-to-code conversion at runtime — that is, nodes representing calls that accept a string argument and cause that string to be interpreted or executed as code by the JavaScript engine or operating system. This pattern covers two sub-categories: (i) function constructor instantiation, in which a NewExpression node with the callee resolving to the built-in Function constructor is detected, and the constructor's string arguments represent dynamically assembled function body code; and (ii) timer functions invoked with string arguments, where a CallExpression targets a built-in timer API and passes a string as the first argument rather than a function reference, causing the string to be parsed and executed as code after the specified delay. In the context of AI configuration file analysis, string-to-code conversion functions are of particular concern because they allow an attacker to encode harmful instructions as string data that evades static pattern matching while remaining executable at agent runtime. Both sub-categories are structurally distinguishable in the AST independently of identifier aliasing, whitespace variation, or comment insertion in the source text.

335 [0029] E. Object-Structure Pattern (329). The object-structure pattern (329) targets ObjectLiteralExpression AST nodes whose property key sets match a configurable signature associated with a known data transfer or exfiltration payload schema. Certain categories of attack embedded in AI configuration files take the form of structured data objects whose shape (property key set) is characteristic of a particular malicious payload class, such as credential bundles, webhook invocation payloads, or API request bodies carrying sensitive fields. The pattern matcher (308) identifies ObjectLiteralExpression nodes, enumerates their PropertyAssignment child nodes to extract the string representation of each property key, and tests the resulting key set against a catalog of known payload signatures. A match is reported when the key set of the observed object literal is a superset of a minimum required key set defined in the pattern catalog (310), and all required keys are present in the correct structural positions within the ObjectLiteralExpression node.

VI. The matchSemanticPattern() Algorithm

350 [0030] The core of the semantic pattern matcher (308) is the matchSemanticPattern() algorithm, which implements a recursive descent traversal of the AST rooted at each source file's root node. The algorithm maintains a visitor stack and a finding accumulator. At each AST node, the algorithm performs the following steps: (1) the node's kind discriminant is compared against the set of AST node types that are targeted by at least one pattern in the pattern catalog (310); (2) if the node kind matches a targeted type, each applicable pattern's matching predicate is evaluated against the node and its immediate children; (3) if a pattern's predicate evaluates to true, a finding object is created and appended to the finding accumulator, carrying the pattern identifier, the severity level defined in the pattern catalog (310), the node's positional metadata translated to source-file line and column numbers, and the context prose string from the enclosing extracted block object (316); (4) regardless of whether a match was found, the algorithm recurses into each child node of the current node, unless the current node is of a leaf type. The recursion terminates naturally at leaf nodes.

VII. Obfuscation Resistance (340, 342, 344)

367 [0031] A principal advantage of the AST structural matching approach (340) over string or regular expression matching (342) is its resistance to obfuscation (344), as illustrated in FIG. 2. When a static analysis tool relies on string matching or regular expressions to detect dangerous patterns, an adversary may defeat detection through any of the following transformations: (a) inserting whitespace characters between tokens, since whitespace is typically significant in regex patterns but irrelevant to the AST; (b) inserting code comments within expressions, since comments are stripped during lexical analysis and do not appear as AST nodes; (c) splitting string literals into concatenated segments, since the string value is represented as a BinaryExpression or TemplateLiteral in the AST rather than as a single raw token; (d) aliasing function or object names through intermediate variable assignments, since the AST preserves the structure of the aliasing expression independently of the identifier names chosen.

382 [0032] Against each of these obfuscation strategies, the AST structural matching approach (340) employed by the present invention is robust: the structural match predicate operates on the node kind and child structure of the AST, which is invariant under whitespace, comment, and syntactic reformatting transformations. Identifier aliasing is addressed through a lightweight data-flow-sensitive identifier resolution pass that, for each Identifier node encountered during traversal, attempts to resolve the identifier to its definition site and substitute the defined value's AST representation before applying the match predicate.

VIII. Cross-Block Correlator (312)

393 [0033] In many attack scenarios, no single fenced code block within a markdown configuration file contains a complete attack chain. Instead, the attack is distributed across multiple code blocks, with each block contributing one stage of the attack. For example, one block may define a utility function for reading environment variables, a second block may define a network transmission function, and a third block may invoke both functions in sequence. The cross-block correlator (312) addresses this attack class by accumulating all findings emitted by the finding emitter (314) across all code blocks in the markdown configuration file, and then applying a set of cross-block combination rules to the accumulated finding set.

405 [0034] In a preferred embodiment, these combination rules are expressed as conjunctive predicates over the finding types: a rule specifies that when a finding of type A and a finding of type B are both present in the accumulated finding set for the same file, a composite finding of type C is generated at a severity level that may be higher than either of the constituent findings individually. This severity escalation mechanism ensures that multi-stage attack chains that individually fall below a reporting threshold are surfaced as high-severity composite findings when the combination meets a known attack chain pattern. The composite finding carries references to the constituent findings, including their respective source positions within the markdown configuration file, enabling security analysts to trace the full attack chain across code blocks.

IX. The analyzeFile() Integration Method

418 [0035] The top-level entry point of the system is the analyzeFile() method, which integrates all stages described above. The method accepts a file system path or content string for a markdown configuration file (300) and returns a structured analysis result comprising: (a) the list of extracted block objects (316) produced by the code block extractor (302); (b) the per-block finding lists produced by the semantic pattern matcher (308) and finding emitter (314); and (c) the composite finding list produced by the cross-block correlator (312). The method coordinates the pipeline by invoking the code block extractor (302) on the raw file content, iterating over the resulting block sequence to invoke the language detector (304), AST parser (306), and semantic pattern matcher (308) for each block, collecting all per-block findings, and finally passing the collected findings to the cross-block correlator (312) before returning the complete result.

433 [0036] This modular architecture enables each component to be tested independently, replaced with an alternate implementation, or extended with additional language parsers or pattern catalog entries without modification to the other components. The pattern catalog (310) is designed to be extensible: new semantic patterns may be added by appending entries to the catalog array, and each entry specifies the target AST node type, the matching predicate, the severity level, and a human-readable description for inclusion in emitted findings. This extensibility ensures that the invention remains applicable to novel attack patterns discovered after the system's initial deployment.

X. Method Steps (FIG. 3, 350–374)

445 [0037] Referring now to FIG. 3, the method of the present invention proceeds as follows. At step (350), the method begins execution. At step (352), the method receives a markdown configuration file associated with an AI CLI platform; the file may be supplied as a file system path, an in-memory string, or a byte buffer. At step (354), the code block extractor (302) applies its finite-state-machine parser to identify and extract all fenced code blocks from the file, producing a sequence of extracted block objects (316) as described above in Section II.

455 [0038] At decision step (356), the method tests whether the current extracted block object carries a language tag in its language field. If a language tag is present (YES branch from decision (356)), the method proceeds directly to AST construction at step (360). If no language tag is present (NO branch from decision (356)), the method proceeds to step (358), where the language detector (304) applies the token-frequency heuristic analysis described in Section III to infer the language of the block, after which the method proceeds to step (360).

465 [0039] At step (360), the AST parser (306) constructs an Abstract Syntax Tree from the block content using the language-appropriate parser frontend described in Section IV. At step (362), the semantic pattern matcher (308) initiates recursive-descent traversal of the constructed AST. At decision step (364), the method tests whether the current AST node during traversal matches a security pattern from the pattern catalog (310). If no match is found (NO branch from decision (364)), the traversal continues to the next AST node, looping back to step (362). If a match is found (YES branch from decision (364)), the method proceeds to step (366), where the finding emitter (314) creates and records a finding object annotated with the source position, pattern type, severity, and context string. After recording the finding, the traversal resumes with the next AST node, looping back to step (362) until all nodes have been visited.

479 [0040] At decision step (368), the method tests whether additional unprocessed extracted block objects remain in the sequence produced at step (354). If additional blocks remain (YES branch from decision (368)), the method loops back to step (354) to process the next block. If no additional blocks remain (NO branch from decision (368)), the method proceeds to step (370), where the cross-block correlator (312) applies cross-block combination rules to the full set of accumulated findings from all blocks. At step (372), the finding emitter (314) emits the complete set of composite findings annotated with references to their constituent per-block findings. At step (374), the method terminates and returns the analysis result to the caller.

XI. Alternative Embodiments

492 [0041] In an alternative embodiment, the system is integrated into a continuous integration pipeline as a pre-commit or pre-merge hook, receiving each modified markdown configuration file as input and blocking the commit or merge if any finding of CRITICAL or HIGH severity is emitted. In this embodiment, the analyzeFile() method is invoked for each staged markdown file, and the exit code of the analysis process is set to a non-zero value when at least one blocking finding is present, causing the CI pipeline to fail and report the finding details to the developer.

503 [0042] In another alternative embodiment, the pattern catalog (310) is stored as a human-readable configuration file rather than as compiled code, enabling security teams to customize the catalog of security-sensitive function names, property access paths, and object-structure signatures without recompiling the analysis system. In this embodiment, the catalog file is loaded and parsed at system startup, and its entries are compiled into an in-memory catalog structure that is passed to the semantic pattern matcher (308). This approach facilitates rapid response to newly discovered attack patterns by enabling catalog updates to be deployed as configuration changes rather than as code changes.

515 [0043] In yet another alternative embodiment, the findings emitted by the system are encoded in Static Analysis Results Interchange Format (SARIF) version 2.1.0, enabling integration with security tooling ecosystems that consume SARIF output including GitHub Advanced Security, Azure DevOps Security, and IDE extensions for Visual Studio Code. Each finding is encoded as a SARIF Result object with a severity level mapped from the pattern's severity field and a location encoded as a SARIF PhysicalLocation referencing the source markdown configuration file, the start line and column within the file, and the end line and column derived from the AST node's positional metadata.

Abstract

695 A system and method analyze fenced code blocks embedded in markdown configuration files used by artificial intelligence coding assistant CLI platforms such as CLAUDE.md, .cursorrules, and AGENTS.md. A code block extractor (302) applies finite-state-machine parsing to extract each fenced code block as a block object (316) comprising language tag, content, start/end line numbers, and surrounding prose context. A language detector (304) resolves unlabeled blocks via token-frequency heuristics. An abstract syntax tree parser (306) constructs a structural AST from each block using a language-appropriate compiler frontend. A semantic pattern matcher (308) traverses the AST by recursive descent, applying match predicates for five security pattern types drawn from a pattern catalog (310): function-call (322), property-access (324), dynamic-import (326), indirect code execution (328), and object-structure (329). Findings are annotated with source position and severity. A cross-block correlator (312) applies combination rules across all blocks to surface composite attack chains. Because matching operates on AST structure rather than raw text, the system is resistant to obfuscation through whitespace variation, comment insertion, and identifier aliasing.
